# 小满 AI 决策引擎官网方案

## Summary
基于现有产品前端的视觉语言，官网第一版采用 `Next.js` 同栈实现，整体风格保持“浅色冷蓝基底 + 克制光晕 + 简约卡片 + 轻科技感”，避免做成重视觉、重动效的炫技页。站点结构采用“官网首页 + 产品详情页”的双层信息架构：首页负责品牌建立、需求背景、核心能力概览、使用路径引导和转化；详情页负责完整产品说明、使用流程拆解和后续可持续补充的业务痛点内容。

## Key Changes
### 1. 视觉与设计系统
- 直接沿用 `xiaoman-core-frontend` 的基础设计语义：浅色背景、冷蓝主色、轻微蓝绿径向光晕、圆角卡片、细边框、简洁按钮和清晰层级。
- 官网单独补一套 landing page token，但不偏离现有产品色系；重点增加：
  - hero 区域的科技感背景层
  - 数据流/决策流样式的视觉模块
  - 更适合营销页面的 section 间距、标题层级和展示卡片
- 动效只做轻量级增强：首屏淡入、卡片轻微浮动、流程线条/光点扫过，不引入高噪声动画。
- 字体和排版继续偏产品化、企业化，不做夸张标题风格，确保“稳重的 AI 科技感”。

### 2. 页面信息架构
- 首页 `/`
  - Hero：一句话定义产品、补充价值说明、主 CTA/次 CTA
  - 需求背景：为什么企业需要 AI 决策引擎
  - 企业痛点：此区块先保留结构与占位，后续替换为你提供的业务文案
  - 核心能力：围绕“训练/发布/推理/运营”或“决策编排/模型接入/规则控制/在线服务”组织
  - 使用引导总览：用 3-5 步完整说明产品如何落地使用
  - 适用场景/客户价值：展示面向哪些业务决策场景
  - CTA：预约演示、联系咨询、查看详情页
- 产品详情页 `/product` 或 `/guide`
  - 产品定位与目标用户
  - 完整产品说明
  - 完整使用流程
  - 模块说明
  - 常见接入方式/实施方式
  - 后续可扩展 FAQ、行业方案、案例区块
- 导航至少包含：首页、产品说明/使用指南、需求背景/痛点锚点、联系或演示入口。

### 3. 内容表达方式
- 第一版以中文为主，但从组件结构和文案组织上为未来双语切换预留能力；不必首版就实现国际化框架。
- 产品说明部分重点讲“是什么、解决什么问题、怎么用、适合谁、上线后如何运营”，避免只有营销语没有操作闭环。
- 使用引导必须是清晰的线性路径，建议按以下逻辑组织：
  1. 明确业务决策场景
  2. 配置输入指标/规则/表单
  3. 接入模型或训练结果
  4. 发布决策服务
  5. 在线调用与结果反馈
- 企业痛点与需求背景区块先保留结构化占位文案，后续只替换文本，不改版式。

### 4. 技术实现边界
- 维持与 `xiaoman-core-frontend` 一致的技术栈：`Next.js App Router + React + Tailwind CSS v4`。
- 优先抽出官网通用组件，而不是直接复用业务后台的 `AppShell/Sidebar` 结构；官网应是更轻的 marketing shell。
- 可以复用现有基础 UI 风格与部分 button/card 组件写法，但官网需要新增：
  - 营销页导航栏
  - Hero/Glow 背景组件
  - 流程步骤组件
  - 能力卡片组件
  - CTA 区块组件
- 首版默认做响应式，重点保证桌面端精致、移动端清晰可读。

## Test Plan
- 桌面端与移动端都能完整浏览首页和详情页，导航与锚点跳转正常。
- 配色、边框、圆角、按钮风格与 `xiaoman-core-frontend` 保持同源，不出现明显“两个产品”的视觉割裂。
- 首页能清楚回答 4 个问题：
  - 这是什么产品
  - 它解决什么问题
  - 企业为什么现在需要它
  - 用户如何开始使用
- 详情页能独立承担“产品说明书”角色，不依赖销售口头解释也能看懂基础使用路径。
- 痛点/需求背景模块在文案未最终敲定时仍保持完整版面，不因占位内容导致布局失衡。

## Assumptions
- 当前 `xiaoman-core-portal` 可以按新项目方式初始化，不需要兼容既有页面结构。
- 第一版不做后台登录、演示系统联动、在线表单提交等复杂交互，CTA 先以按钮入口和联系占位为主。
- 详情页路由名称可在实施时二选一：更偏营销则用 `/product`，更偏说明书则用 `/guide`；默认推荐 `/product`。
- 双语预留只做结构预留，不在首版交付中强制实现完整 i18n。
- 企业痛点、需求背景、部分场景价值文案后续由你补充时，只替换 copy，不推翻本次页面骨架。
